FoodDB 🍎

Local-first personal nutrition facts catalogue — SQLite, Typer CLI, and Streamlit UI sharing one service layer, with a regex label parser and swappable OCR backends.

GitHub · Back to Projects

  • Problem: Tracking nutrition facts across different stores and package sizes requires a tool that stores data once but is queryable from anywhere — CLI for quick adds, UI for browsing.
  • Approach: Layered architecture (ORM → repository → services → dual CLI/UI surfaces); regex-based label parser with optional OCR for image imports; swappable OCR backends behind a clean interface.
  • Outcome: A self-contained, portable local tool with consistent behavior across both surfaces, typed error handling, and an extension-friendly design.

FoodDB is a personal utility project that demonstrates software engineering discipline applied to a real problem. The CLI and Streamlit UI are both thin clients over the same repository and service layer — they share one SQLite file, meaning CLI writes are immediately visible in the UI after a page refresh, with no sync required.

Core capabilities:

  • Full CRUD for foods with complete nutrition facts (macros, micros, ingredients, best-by date)
  • Store variants — track the same product at Costco vs. Aldi with price and package sizing
  • Inventory tracking per variant (quantity on hand)
  • Import from label text (paste) or label image (upload): regex parser extracts structured fields; OCR converts image → text → parser
  • Fuzzy search with token overlap + SequenceMatcher scoring; filterable by category, brand, store
  • Export to JSON or CSV
  • Unit conversion table with UI calculator

No HTTP server. Both surfaces call the same Python service layer directly. The DB path comes from a single pydantic-settings config object — change it in one place and both surfaces follow.

Engineering Notes


Design Decisions
  • ORM / Pydantic separation — SQLAlchemy ORM classes map to DB tables; Pydantic schemas are the API layer. No ORM objects leak above the repository boundary.
  • foods vs food_variants — canonical nutrition facts live on the food; store/price/package details live on variants. Avoids duplicating nutrition data across stores.
  • ParsedNutrition.warnings list — explicit, inspectable signal for uncertain extractions. The UI surfaces warnings before saving so the user can correct them, rather than silently saving wrong data.
  • Lazy OCR imports — the package works without Tesseract or pytesseract installed. A clear RuntimeError with install instructions is raised if OCR is called without extras.
OCR Backend Swapping
  • Tesseract (default): grayscale → bilateral filter → Otsu threshold → morphology open → pytesseract. HEIC/HEIF handled via pillow-heif → temp PNG.
  • Ollama vision (alternative): image → PNG bytes → base64 → multimodal chat completion via OpenAI-compatible API. Switch with OCR_BACKEND=ollama_vision in .env — no code change required.
  • Typed error handling: OCRException with a typed OCRErrorType enum. The import page translates each error type into an actionable user-facing hint.
Fuzzy Search
  • Normalizes query and candidate (lowercase, & → and, collapse whitespace)
  • Returns 1.0 for direct substring match
  • Otherwise: 0.65 × token_overlap + 0.35 × SequenceMatcher_ratio
  • Token overlap handles multi-word queries; SequenceMatcher handles misspellings and OCR noise
  • Thresholds configurable via .env without code changes
Config & Tooling
  • pydantic-settings for all configuration — .env support is automatic; no scattered os.getenv()
  • uv for dependency and environment management
  • Streamlit @st.cache_resource ensures a single DB connection per server process
  • Stable Streamlit widget keys via make_form_key(action, entity_id) to avoid form key collisions across dynamic renders
